
## Why a workspace (not a single dual-target crate)

The deciding constraint: `rand` on `wasm32-unknown-unknown` needs `getrandom`'s `js` feature, which drags in `wasm-bindgen` and is known to break non-web targets. Keeping that feature — and the wasm-bindgen/serde glue — in a separate `wasm` crate that depends on `core` keeps the native/TUI build clean and free of web-only dependencies. `core` stays a plain Rust library that also ships the TUI binary; `wasm` is a thin adapter.

## Migration map (deferred to the scaffolding session)

- `src/lib.rs` → `crates/core/src/lib.rs`
- `src/bin/main.rs` → `crates/core/src/bin/main.rs`
- `tests/simple.rs` → `crates/core/tests/simple.rs`
- root `Cargo.toml` → becomes the **workspace** manifest; `core` gets its own `crates/core/Cargo.toml` carrying the current dependencies.
- **`clap` must stop being a plain dependency of `core`**, or it gets compiled into the wasm build via `core`. Options: (a) make `clap` an optional dependency behind a `cli` feature that also gates the `bin` target, or (b) split the TUI into its own `crates/cli` binary crate. Decide during scaffolding, but option (a) is _much more preferred_; the README lists `clap` as TUI-only on that assumption.

## Crate rename

The crate is currently named `cli_90` (referenced as `cli_90::` in `src/bin/main.rs` and `tests/simple.rs`). Part of the restructuring work includes renaming to `ninety_ci_core` for clarity now that "cli" is only one of two front-ends. This touches those two `use` sites and the manifest. Recommended, not required.
