# Repository Map

A high-level guide to where things live in the `90ci` codebase, for developers and code reviewers. Config files and documentation are intentionally omitted; this map covers the code: UI, designs, Rust, the WASM wrapper, and tooling.

## Birds-eye layout

The repo is a hybrid Rust + TypeScript monorepo. A Rust workspace (`crates/`) holds the Monte-Carlo simulation engine and a WASM wrapper that compiles it for the browser. A Vite + React app (`web/`) is the front-end, which loads the WASM module to run simulations client-side. Design mockups live in `design/`, and the single build script lives in `scripts/`.

```
crates/        Rust workspace: core engine + WASM wrapper
  core/        ninety_ci_core тАФ Monte-Carlo CI library (+ optional CLI binary)
  wasm/        cdylib that exposes core over the JSтЖФWASM boundary
web/           Vite + React + TypeScript front-end
  src/         UI source (components, hooks, model types, styles)
  src/wasm/    generated wasm-pack output (git-ignored, built on demand)
design/        UI mockups, wireframes, and design-direction prototypes
scripts/       build automation
```

The data flow is one direction: React UI тЖТ `web/src/wasm` glue (generated) тЖТ `crates/wasm` FFI тЖТ `crates/core` engine, and the `Simulation` result flows back out the same path.

## Front-end UI (`web/`)

React + TypeScript, built with Vite. The app is a two-panel calculator: a model editor on the left, a result panel on the right.

Entry & app shell:
- `web/index.html` тАФ HTML entry point; mounts the root div and loads `main.tsx`.
- `web/src/main.tsx` тАФ React bootstrap; renders `<App>` into the root.
- `web/src/App.tsx` тАФ root component; owns model and result state; lays out `TopBar`, `ModelEditor`, and `ResultPanel`. On a run error it clears `result` so a stale chart never sits beside the error banner (E-10), and it disables Run while the engine has failed to load (`runDisabled`, E-09).
- `web/src/model.ts` тАФ shared TypeScript types and constants: `Shape`, `Variable`, `Model`, `SimResult`, `SHAPES`. The sample count is fixed at 5,000 (PRD ┬з1), so `Model` carries no `samples` field; `SimResult.samples` is echoed back by the engine. `Variable.p5`/`p95` are `number | null` тАФ `null` represents a blank bound, kept distinct from a real `0` so an empty cell is never silently coerced (E-04).
- `web/src/validation.ts` тАФ client-side model validation (Stage 4, PRD ┬з3). `firstValidationError(model)` reports one violation at a time (mirroring the engine's one-per-run precedence); today it covers blank bounds (E-04), naming the offending variable and which bound: "`X`: enter a number for the 5th/95th bound." `hasBlankP5`/`hasBlankP95` are per-field predicates the editor uses to mark cells; `parseBound` maps a raw input string to `number | null` (blank тЖТ `null`, non-finite тЖТ `null`). Shared by the hook (to block the engine call) and `ModelEditor` (to parse/mark cells) so the rule lives in one place.
- `web/src/vite-env.d.ts` тАФ Vite ambient type declarations.

Components (`web/src/components/`):
- `TopBar.tsx` тАФ breadcrumb header ("Models / [name]").
- `ModelEditor.tsx` тАФ left panel: equation input, variable management, Run button. (Sample count is fixed at 5,000; there is no sample-count control.) Parses a blank input to `null` instead of `0` via `parseBound` from `validation.ts` (E-04); blank bound cells get a `num-input--invalid` marker driven by the `validationError` prop (the client validation channel only, never engine/init errors). Accepts optional `runDisabled` (E-09) and `validationError` props. **A11y / distribution semantics (Stage 6, PRD ┬з5 + ┬з7):** every input carries an `aria-label` (title тЖТ "Model name", formula тЖТ "Model equation", per-row name/distribution/5th/95th labels include the variable name so they are unambiguous). Each variable row has a focusable `.info-btn` (`aria-label="Distribution semantics"`) carrying `aria-describedby` to a `role="tooltip"` span exposing the exported `DISTRIBUTION_TOOLTIP` copy тАФ the single honest ┬з5 string ("For normal, these are the ~5th/95th percentiles. For uniform and range, they are the full minimum and maximum тАФ the middle 90% falls inside them."). The `aria-describedby` wiring puts the copy in the button's accessible *description* (its name stays "Distribution semantics") so screen-reader users reach it, while CSS reveals the same span visually on hover **and** focus тАФ reachable both ways, and it makes no percentile claim for uniform/range. The remove-variable button carries an `aria-label` too. Column headers stay "5th"/"95th" and the percentile-bounds hint note is unchanged.
- `ResultPanel.tsx` тАФ right panel: renders results, or the empty/error state. The error banner carries `role="alert"` (PRD ┬з3 / ┬з7 a11y).
- `CIHero.tsx` тАФ headline display of the 90% confidence interval bounds and width.
- `OutputChart.tsx` тАФ SVG histogram of the output distribution with the 90% CI band shaded and two dashed `ciLow`/`ciHigh` markers carrying numeric labels (PRD ┬з6). A zero-width / degenerate model (`buckets.length < 2`, E-03b) renders a visible single-spike marker with its value instead of a blank `<svg/>`. Very-narrow spreads enforce a min marker separation and collapse the two edge labels into one centred label so they stay legible тАФ rendered as a `loтАУhi` range when the bounds differ so neither is silently dropped. Bucket widths come from the engine (no front-end `step`).
- `Sparkline.tsx` тАФ miniature distribution preview per variable (bell for normal, plateau for uniform/range).

Hooks & WASM glue (`web/src/`):
- `hooks/useNinetyCi.ts` тАФ hook that initializes the WASM module, invokes `simulate`, and tracks loading/error state; maps UI percentiles to engine bounds and shapes the output into `SimResult`. Passes a hard-wired `SAMPLES = 5_000` to the engine (PRD ┬з1). Runs `firstValidationError` before calling the engine and, on a violation, sets a dedicated `validationError` (a channel separate from engine/init `error`, so an inline blank-bound cell marker never fires under an unrelated failure) and does **not** call `wasmSimulate` (E-04). Tracks engine init via `engineReady` (`null` pending / `true` loaded / `false` failed); an `init()` rejection sets a persistent "Couldn't load the calculator engine тАФ reload the page." error and `engineReady === false`, which `App` turns into `runDisabled` (E-09). Calls the engine's 3-arg `simulate(equation, vars, SAMPLES)` тАФ the bucket `step` is no longer a front-end constant; `core` computes it from the output magnitude (PRD ┬з6).
- `web/src/wasm/` тАФ wasm-pack output (ES module + `.d.ts` bindings). Generated by the build script and git-ignored; do not hand-edit.

Styles:
- `web/src/styles/app.css` тАФ global CSS: design tokens (color, type, spacing) plus component styles. Includes `.info-btn`/`.info-tooltip` styles for the ┬з5 distribution-semantics tooltip (revealed on `:hover`/`:focus`/`:focus-visible`) and the `.num-input--invalid` cell marker. Three text-color tokens were nudged to meet WCAG AA against their backgrounds (Stage 6, ┬з7): `--ink-3` `#9b988f`тЖТ`#737068` (4.54:1), `--blue` `#2a72d4`тЖТ`#2065bb` (4.88:1), `--grn` `#4f8a5b`тЖТ`#477851` (4.55:1).

## Rust core engine (`crates/core`)

`ninety_ci_core` тАФ the Monte-Carlo confidence-interval library. Pure Rust, no browser dependency. An optional `cli` feature builds a TUI binary.

- `crates/core/src/lib.rs` тАФ the engine. Key items: `Distro` (Normal / Uniform / DiscreteUniform sampling); `Equation<Status>` тАФ a phantom-typed state machine (`UnderDefined` тЖТ `FullyDefined` тЖТ `Evaluated`) that extracts variables, samples distributions, evaluates the equation per sample, builds a histogram, and computes the 90% CI; `VariableDescription` (input) and `Simulation` (output) types; `simulate()` entrypoint and the `ci90()` convenience wrapper (both 3-arg: `eq, vars, iterations` тАФ the bucket `step` is no longer a parameter). `compute_step(lowest, highest)` derives the histogram bucket width from the observed output range, targeting ~50тАУ100 buckets (divisor 75; degenerate `highest <= lowest` тЖТ a positive default of `1.0` yielding a single-bucket spike), and clamps the result to a magnitude-tied positive minimum so it can never underflow to a step that fails to advance the bucket loop; `evaluate()` calls it on the finite series before `compute_histogram` (which still takes an explicit `bucket_size`). `compute_histogram`'s bucket-building loop carries a no-progress guard (`if next == bucket break`) as defense-in-depth so a subnormal step can never hang it. `evaluate()` filters out non-finite per-sample outputs (┬▒inf / NaN from degenerate models such as division by zero) so they never reach `compute_histogram`; when no finite samples survive, `simulate` returns a human-readable `Err` instead of trapping (E-07). `compute_histogram` sorts with `total_cmp` and clamps bucket indices to absorb float rounding at the range extremes; `ninety_ci` normalizes by the surviving finite-sample count (not the configured resolution) and propagates an `Err` rather than unwrapping on a degenerate histogram. **Authored-model validation (Stage 3, ┬з4 error matrix)** reports one violation per run, in this precedence: empty/blank equation тЖТ `"Enter an equation."` (E-05a); empty vars тЖТ `"Add at least one variable."` (E-05b); duplicate variable names, scanned in `add_variables` before the `HashMap` can silently merge them тЖТ `"Two variables are named \`X\` тАФ names must be unique."` (E-12); then per variable in `add_variable`: a name not referenced in the equation тЖТ `"\`X\` is defined but not used тАФ use it or remove the row."` (E-02), strictly-inverted bounds `lower > upper` (equal bounds stay valid, e.g. `range(0,0)`) тЖТ `"\`X\`: 5th (lo) must be below 95th (hi)."` (E-03), unsupported shape тЖТ the `Distro::new` guard `"Unsupported distributionтАж"` (E-11); finally, equation tokens with no variable row, named individually тЖТ `"\`X\` is used in the equation but not defined тАФ add a variable row for it (or remove it)."` (E-01, with a comma-joined multi-token variant). **RNG injection (testability):** sampling takes an injected `&mut dyn RngCore` threaded through `sample_variable тЖТ add_variable тЖТ add_variables`; the public `simulate`/`ci90` are unchanged and delegate to a private `simulate_with_rng(eq, vars, iterations, rng)` passing `thread_rng()`, while tests pass a seeded `StdRng` to make the Monte-Carlo CI reproducible (so numeric assertions are deterministic, not flaky). Includes the unit-test module, whose `simulate_normal_seeded_recovers_ci_deterministically` seeds the engine and tightly asserts the recovered [100,200] CI.
- `crates/core/src/bin/main.rs` тАФ CLI binary, gated behind the `cli` feature. Clap parser for `--equation` and repeatable `--var name,distribution,lower,upper`; prints the computed 90% CI.
- `crates/core/tests/simple.rs` тАФ integration tests over the public API. The `normal` case asserts only seed-INDEPENDENT invariants (finite, ordered `ci_low < ci_high`, `samples == 5000`, parallel `buckets`/`counts` lengths, non-empty buckets, a generous `[0, 300]` sanity window) so it cannot flake against the unseeded `thread_rng`; the tight numeric recovery is the seeded unit test in `lib.rs`. The `uniform` case checks `1 + uniform[1,2]` recovers `[2, 3]` within `0.1` (stable: the uniform 5th/95th estimator has a tiny SE at N=5000).

## WASM wrapper (`crates/wasm`)

A thin `cdylib` that exposes `ninety_ci_core` to JavaScript via `wasm-bindgen`. This is the only FFI boundary.

- `crates/wasm/src/lib.rs` тАФ the FFI layer. `VarInput` (owned strings deserialized from JS) and `SimOutput` (serialized back to JS); `init()` installs `console_error_panic_hook` so Rust panics surface in the browser console; `simulate(equation, vars, iterations)` borrows `VariableDescription`s from the inputs, calls the core engine, and returns `{ ci_low, ci_high, buckets, counts, samples }`. (The bucket `step` is computed engine-side and is no longer a boundary parameter тАФ PRD ┬з6.)
- `crates/wasm/tests/boundary.rs` тАФ WASM boundary harness (`wasm-bindgen-test`, `run_in_browser`): a round-trip test that calls `simulate` over the built glue and asserts a plausible `SimOutput`, plus a divide-by-zero test asserting that a degenerate model returns a readable error string (never a wasm `"unreachable"` trap; E-07); and a constant-output test asserting the single-bucket spike path marshals back as `buckets.len() == 1` (Stage 5, ┬з6). Stage 3 adds one test per ┬з4 validation row (E-05a, E-05b, E-12, E-02, E-03, E-11, and both the single- and multi-token E-01 branches) asserting the thrown JS string names the offending token/row and carries the corrective phrasing; a header comment documents the one-violation-per-run precedence. Run with `wasm-pack test --headless --chrome crates/wasm`.
- `crates/wasm/webdriver.json` тАФ ChromeDriver capabilities (`--no-sandbox`, `--headless`, `--disable-gpu`, `--disable-dev-shm-usage`) so the boundary test launches Chrome in headless/sandboxed environments.
